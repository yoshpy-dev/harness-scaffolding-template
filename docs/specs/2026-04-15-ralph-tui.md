# Ralph Status TUI — Lazygit 風インタラクティブダッシュボード

## 概要

`ralph status` コマンドを Lazygit 風の TUI (Terminal User Interface) に拡張する。Go + Bubble Tea フレームワークで実装し、スライス一覧・詳細・依存関係グラフ・ログの4ペインレイアウトを提供する。vim スタイルのキーバインド (h/j/k/l) でペイン間・項目間を移動し、stuck/failed スライスの再試行やアボートなどの操作を Space キーで実行できる。

## 背景と課題

### 現状

`ralph status` は静的なテーブル出力を提供する。`--watch` オプションで定期的にリフレッシュできるが、以下の制限がある:

- 出力は一方向（表示のみ）で、操作は別ターミナルから `ralph abort` 等を手動実行する必要がある
- ログの確認には別途ファイルを開く必要がある
- 依存関係の可視化がない
- stuck/failed スライスの再試行には手動でのコマンド実行が必要

### 理想状態

- 一つの TUI 画面でスライスの状態確認・ログ追跡・操作（再試行・アボート）を完結できる
- リアルタイムでステータスとログが更新される
- vim ユーザーに馴染みのあるキーバインドで直感的に操作できる
- 依存関係グラフが視覚的に表示され、ブロッキング関係が一目で分かる

## 要件

### 機能要件

- [ ] **4ペインレイアウト**: スライス一覧（左上）、詳細（右上左）、依存関係グラフ（右上右）、ログ（右下）を表示する
- [ ] **アクションパネル** (左下): 選択スライスに対する操作一覧を表示する
- [ ] **プログレスバー** (最下部): 完了率・ETA・完了数/総数を表示する
- [ ] **vim キーバインド**: h/l でペイン移動、j/k でペイン内移動、Space でアクション実行
- [ ] **Tab/Shift+Tab**: ペインの順送り・逆送り
- [ ] **リトライ操作 (`r`)**: stuck/failed スライスに対して `ralph-pipeline.sh --resume` を再実行する
- [ ] **アボート操作 (`a`)**: 選択スライスを確認ダイアログ付きでアボートする
- [ ] **全アボート操作 (`A`)**: 全スライスを確認ダイアログ付きでアボートする（`ralph abort` 相当）
- [ ] **ログ表示 (`L`)**: 選択スライスのフルログをページャー (`$PAGER` or `less`) で表示する
- [ ] **エディタで開く (`e`)**: 選択スライスの worktree を `$EDITOR` で開く
- [ ] **依存フォーカス (`d`)**: 依存関係ペインにフォーカスを切り替える
- [ ] **ヘルプ (`?`)**: 全キーバインドをオーバーレイで表示する
- [ ] **フィルタ (`/`)**: スライス名でフィルタリングする
- [ ] **リアルタイム更新**: fsnotify でステートファイルを監視し、変更時に自動更新する
- [ ] **ログストリーミング**: 選択スライスのログ末尾を自動追従する (tail -f 相当)
- [ ] **ターミナルリサイズ対応**: ウィンドウサイズ変更時にレイアウトを自動調整する

### 非機能要件

- [ ] TTY 検出: TTY でなければ TUI を起動せず、既存のテーブル出力にフォールバックする
- [ ] `--no-tui` フラグ: 明示的に既存のテーブル出力を使用する
- [ ] `--json` フラグ: 既存の JSON 出力を維持する（TUI に影響しない）
- [ ] 単一バイナリ: `go build` で単一実行ファイルを生成する
- [ ] ビルド不要時のフォールバック: Go バイナリが存在しない場合、既存のシェルスクリプト出力を使用する
- [ ] 起動時間: 500ms 以内に TUI を描画する
- [ ] メモリ使用量: 50MB 以下（大量のログファイルがあっても）

## 受け入れ基準

- [ ] `ralph status` を TTY で実行すると TUI が起動すること
- [ ] 4ペインレイアウトがターミナルサイズに応じてリサイズすること
- [ ] h/j/k/l でペイン間・項目間を移動できること
- [ ] Space キーでアクションを実行できること
- [ ] stuck/failed スライスに対して `r` で再試行が実行されること
- [ ] `a` でスライスのアボートが確認ダイアログ付きで実行されること
- [ ] ログペインが選択スライスのログをリアルタイムでストリーミングすること
- [ ] 依存関係ペインがスライス間の依存関係をツリー表示すること
- [ ] プログレスバーが完了率と ETA を表示すること
- [ ] `ralph status --json` は既存の JSON 出力を維持すること
- [ ] `ralph status --no-tui` で既存のテーブル出力にフォールバックできること
- [ ] `?` キーで全キーバインドのヘルプオーバーレイが表示されること

## ユーザーストーリー

1. Ralph Loop ユーザーとして、1つの画面でスライスの進捗を確認し、問題があればその場で対処したい。なぜなら、複数ターミナルを行き来するのは非効率だから。
2. Ralph Loop ユーザーとして、stuck になったスライスを即座に再試行したい。なぜなら、手動でコマンドを組み立てて実行するのは手間がかかるから。
3. Ralph Loop ユーザーとして、各スライスのログをリアルタイムで確認したい。なぜなら、問題の早期発見と原因特定に必要だから。
4. Ralph Loop ユーザーとして、依存関係を視覚的に確認したい。なぜなら、どのスライスがブロックされているかを素早く把握したいから。

## 制約条件

### スコープ内

- Go バイナリ `ralph-tui` の新規作成（`cmd/ralph-tui/`）
- `scripts/ralph` の `status` サブコマンドへの TUI 統合
- 既存ステートファイルの読み取り（書き込みは既存 CLI 経由）
- `go.mod` / `go.sum` の追加
- ビルドスクリプト（`scripts/build-tui.sh` or Makefile target）
- キーバインドの設定ファイル対応（将来拡張向け構造のみ）

### スコープ外

- Web UI の提供
- パイプラインロジックの変更（TUI は監視・制御のみ）
- 新しいステートファイル形式の導入
- `ralph run` / `ralph abort` の Go 完全移植（TUI からは既存シェルスクリプトを呼び出す）
- CI/CD 環境での TUI 実行（ヘッドレス環境では `--no-tui` / `--json` を使う）
- カスタムテーマ / カラースキーム設定

## 調査結果

### コードベース分析

- `scripts/ralph` (339行): POSIX sh で書かれた CLI エントリポイント。`status` サブコマンド (133-176行) がメインの対象。
- `scripts/ralph-status-helpers.sh` (536行): テーブル描画・色制御・プログレスバー・ETA 計算・JSON 出力の全ヘルパー関数。TUI のデータ取得ロジックの参考になる。
- ステート管理: `.harness/state/orchestrator/` (orchestrator.json, slice-*.status, slice-*.log) と `.harness/state/pipeline/` (checkpoint.json, execution-events.jsonl) の2層構造。
- 既存の JSON 出力形式 (`ralph-status-helpers.sh` 455-536行) は TUI のデータモデルのベースとなる。

### ベストプラクティス

- **Lazygit** (Go + Bubble Tea): 分割ペイン TUI の成熟した参考実装。Elm アーキテクチャ (Model → Update → View)。
- **Bubble Tea v2**: 最新版は `tea.Program` のライフサイクル管理が改善。`tea.Cmd` による非同期イベント処理。
- **Lip Gloss v2**: Flexbox ライクなレイアウトエンジン。ペイン分割やボーダー描画に適する。
- **fsnotify**: クロスプラットフォームのファイル監視。macOS (kqueue) / Linux (inotify) をサポート。

### 検討した代替案とトレードオフ

| 選択肢 | メリット | デメリット | 採用 |
|--------|---------|-----------|------|
| Go (Bubble Tea) | Lazygit と同じ技術。分割ペイン・vim 操作・リアルタイム更新を完全サポート。単一バイナリ配布。 | Go ツールチェーンが必要。ビルドステップが増える。 | YES |
| Python (Textual) | 表現力が高い。16.7M 色サポート。 | Python ランタイム依存。この scaffold は言語非依存を指向。 | NO |
| Pure Bash + gum/fzf | 依存最小。既存シェルスクリプトと統一。 | 分割ペインやリアルタイム更新が実現困難。Lazygit ライクな UX は不可能。 | NO |
| Rust (ratatui) | 高パフォーマンス。単一バイナリ。 | Rust ツールチェーンのオーバーヘッド。Bubble Tea ほどのエコシステムがない。 | NO |

## セキュリティ考慮事項

- TUI はステートファイルを読み取り専用でアクセスする。書き込み操作は既存の `scripts/ralph` CLI を `exec.Command` で呼び出す。
- アボート・リトライなどの破壊的操作は確認ダイアログを表示してから実行する。
- `$EDITOR` や `$PAGER` の呼び出し時は引数のサニタイズを行う。
- ログファイルの内容をそのまま表示するため、ANSI エスケープシーケンスインジェクションのリスクがある → ログ表示時はエスケープシーケンスをフィルタリングする。
